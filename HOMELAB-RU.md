# Гибрид: облачная прод-платформа + дачный homelab

> Дополнение к [`PLATFORM-RU.md`](PLATFORM-RU.md). Здесь описана **дачная часть** и её интеграция с облачной платформой.

## 0. Концепция

Два независимых K3s-кластера, объединённых через WireGuard в один административный контур, с разными ролями:

| Сторона | Роль | Что крутится | Аплинк |
|---|---|---|---|
| **Облако** (Selectel/Yandex Cloud) | **Прод** | Клиентские проекты (ФлорМажор и т.д.), публичные домены, входящий HTTPS | Белый IP, бизнес-SLA |
| **Дача** | **Homelab + бэкап-партнёр** | Пет-проекты, dev/staging, холодный бэкап облака, зеркало образов, личные сервисы (Vaultwarden, Gitea, Nextcloud, медиа) | Серый IP, входящий не нужен |

Принципы:
- **Изоляция отказов**: если дача упала — клиентские сайты в облаке живы; если облако режут — пет-проекты на даче живут.
- **Облако никогда не зависит от дачи** в горячем пути.
- **Дача может работать в air-gapped режиме** (нет интернета — пет-проекты в локалке всё равно отвечают).
- **Бэкапы в двух географиях по умолчанию**.

```
                   Internet
                      │
              ┌───────▼────────┐
              │  Cloud K3s      │   ← клиентские прод-сайты, публичный HTTPS
              │  PLATFORM-RU.md │
              └───────┬────────┘
                      │  WireGuard mesh (10.10.0.0/24)
                      │  initiated FROM dacha
              ┌───────▼────────┐
              │  Dacha K3s      │   ← homelab, dev, бэкап-таргет
              │  HOMELAB-RU.md  │
              └────────────────┘
```

## 1. Дачная сторона: железо

### 1.1. Стартовая конфигурация (рекомендую)

| Компонент | Что | Цена б/у | Цена новое |
|---|---|---|---|
| Узел кластера | **2× Lenovo ThinkCentre M920q Tiny / M75q-2** (i5-8500T / Ryzen 5 PRO, 32GB RAM, 1TB NVMe + 1TB SATA) | 35–50 тыс. ₽ за два | 80–100 тыс. ₽ |
| Узел для NAS / бэкапов | **1× самосбор mini-ITX**: AM4, 32GB ECC (если поддерживает), 4× 4TB CMR HDD в ZFS RAID-Z2 | 60–80 тыс. ₽ | — |
| UPS | **APC Smart-UPS 1500VA** (б/у с новой батареей) | 15–25 тыс. ₽ | 35–45 тыс. ₽ |
| Управляемый коммутатор | **Mikrotik CRS305-1G-4S+IN** (с SFP+ для будущих 10G) или **CSS610-8G-2S+IN** | 10–15 тыс. ₽ | — |
| Маршрутизатор/файрвол | **Mikrotik hAP ax2 / RB5009** или **самосбор pfSense на N100** | 8–20 тыс. ₽ | — |
| Стойка/корпуса/кабели/провода | — | ~10 тыс. ₽ | — |
| **Итого** | | **~140–180 тыс. ₽** | **~250 тыс. ₽** |

**Почему так:**
- 2 Tiny + 1 NAS = K3s HA (2 control-plane по embedded etcd работает, но лучше 3; для homelab ок) + большое холодное хранилище отдельно от compute.
- Lenovo Tiny — тихие, малопотребляющие (15–35 Вт под нагрузкой), компактные. Идеальны для домашнего стека.
- Самосбор NAS на ZFS RAID-Z2 даёт ~8 ТБ полезного объёма с устойчивостью к падению 2 дисков.
- Mikrotik CRS305 — задел на 10G линковку между нодами в будущем.
- UPS на 1500 VA удерживает весь стек (~150 Вт) ~30–60 минут — хватит на graceful shutdown через NUT.

### 1.2. Минимум для старта (если бюджет режет)

- 1× Lenovo M920q Tiny (32GB / 1TB), single-node K3s с local-path storage.
- UPS APC Back-UPS 950 VA (~10 тыс. ₽).
- Любой роутер от провайдера + USB-Ethernet адаптер если нужен второй порт.
- **~50 тыс. ₽**, но без HA и без серьёзного хранилища.

### 1.3. Размещение
- **Не сам гараж**, а утеплённая выгородка / шкаф с пассивной или активной вентиляцией.
- Температура зимой ≥ 5°C, летом ≤ 30°C. Для гаража — обязательно утепление + кондиционер или хотя бы вентилятор + дренаж от конденсата.
- Влагопоглотители (силикагель) по углам стойки.
- Дымовой датчик с уведомлением на телефон.

## 2. Сеть и подключение к облаку

### 2.1. Локальная сеть на даче
- VLAN 10 — management (Mikrotik, IPMI, UPS NUT).
- VLAN 20 — кластер (K3s nodes, 10.20.0.0/24).
- VLAN 30 — гостевой/IoT (изолирован от кластера).
- DHCP на Mikrotik, DNS — Pi-hole / AdGuard Home в контейнере на одной из нод.

### 2.2. WireGuard к облаку
**Дача всегда инициирует соединение** наружу (серый IP не помеха):
- На облачной стороне: WireGuard сервер в namespace `wg-mesh`, экспонирован через NodePort или LoadBalancer на UDP/51820.
- На дачной стороне: WireGuard клиент в DaemonSet или systemd-юните на Mikrotik (Mikrotik умеет WG нативно — рекомендую так, чтобы туннель не зависел от состояния K3s).
- Подсеть туннеля: `10.10.0.0/24`. Облако — `10.10.0.1`, дача — `10.10.0.2`.
- На облачной стороне маршрутизация: разрешить из `10.10.0.0/24` доступ к namespace `dacha-mirror` и админ-эндпоинтам Argo CD/Velero/Vault через NetworkPolicy.

### 2.3. DNS
- Внешний домен: `cloud.<your-domain>.ru` — обслуживается ExternalDNS из облачного K3s.
- Внутренний: `lab.<your-domain>.local` — резолвится только в туннеле и в локалке дачи. Pi-hole + кастомные A-записи.

## 3. Дачный кластер: стек

Лёгкая версия облачного PLATFORM-стэка, без излишеств:

| Облако | Дача | Зачем |
|---|---|---|
| K3s | K3s | то же самое |
| Longhorn | **local-path** + **NFS из NAS** | дёшево, на 2-3 нодах достаточно |
| MinIO Operator | **single-node MinIO** | один tenant, для бэкапов и образов |
| Traefik + cert-manager | **Traefik** (только для VPN/локалки) | LE через DNS-01 (cloudflare/yandex) для `*.lab.<your-domain>.ru` или самоподписанные |
| ExternalDNS | не нужно | ручные DNS-записи в Pi-hole |
| Vault | **Vault standalone** | один экземпляр, raft-storage |
| Kubero | Kubero (опционально) | если хочешь push-to-deploy для пет-проектов |
| Argo CD | Argo CD | GitOps |
| Prometheus + Loki | Prometheus + Loki (легче) | мониторинг + remote_write в облако (см. ниже) |
| Velero | Velero | бэкапы дачных pet-проектов в дачный MinIO |

**Что специфичного добавляется:**
- **Harbor** или **Gitea с container registry** — зеркало для критичных образов на случай блокировки Docker Hub / GHCR.
- **Gitea** — зеркало приватных репозиториев (master в GitHub, mirror на даче).
- **Nextcloud / Vaultwarden / Jellyfin / Home Assistant** — личные сервисы.
- **NUT** (Network UPS Tools) — мониторинг UPS, graceful shutdown по сигналу low-battery.

## 4. Бэкап-партнёрство (ключевая фича гибрида)

Облако и дача страхуют друг друга по данным:

### 4.1. Облако → Дача (горячий бэкап на холодное хранилище)

Что реплицируется:
- **Velero бэкапы** облачного K3s (state + PV) в облачный MinIO → асинхронная репликация в дачный MinIO через `mc mirror` или **MinIO bucket replication**.
- **WAL-G архивы** Postgres всех клиентских Supabase-инстансов в облачный S3 → cron-job на даче `mc mirror --watch` или периодическая синхронизация.
- **Дамп Vault** (raft snapshot) раз в сутки → encrypted, в дачный MinIO.

Расписание: непрерывная репликация для S3, ежесуточный snapshot для Vault.

### 4.2. Дача → Облако (опционально)

Что реплицируется:
- **Velero бэкапы** дачного кластера → в выделенный bucket в облачном MinIO. Если дача сгорит — пет-проекты восстанавливаются в облако.
- **Дамп Pi-hole, Home Assistant конфигов**.

Расписание: ежесуточно.

### 4.3. Восстановительные процедуры

Готовые runbook'и в `docs/runbooks/`:
- `cloud-down-restore-to-dacha.md` — поднять копии клиентских сайтов на даче в read-only режиме (для самих, не для клиентов; статус-страница).
- `dacha-down-rebuild.md` — закупить новое железо, восстановить из облачного бэкапа.
- `single-project-pitr.md` — PITR Postgres конкретного проекта из WAL-G.

Тестировать раз в квартал (полное восстановление одного проекта в staging-namespace).

## 5. Failover-сценарии

### 5.1. «Облако упало»
- Клиентские сайты лежат (это прод, выбора нет — поднимаем в облаке у того же или другого провайдера из бэкапа).
- На VPS-«фронтэнде» (отдельная маленькая VM в другом облаке за 200 ₽/мес) поднимается **статус-страница** с информацией. Конфиг лежит в Git, поднимается за 10 минут.
- Личные пет-проекты на даче работают как обычно.
- WireGuard-туннель в этом сценарии не нужен (нечему туннелироваться).

### 5.2. «Дача упала» (свет/интернет/железо)
- Клиентские сайты в облаке не замечают.
- Пет-проекты на даче недоступны до восстановления.
- Бэкапы в облаке продолжают накапливаться, мониторинг шлёт алерт «дача offline > 1 ч».

### 5.3. «Туннель упал» (между облаком и дачей)
- Прод не страдает.
- Репликация бэкапов в дачный MinIO ставится на паузу, при восстановлении догоняет.
- Алерт «replication lag > 6 ч».

## 6. Дорожная карта (после запуска облачной части)

Подразумевается, что Фазы 1–5 из `PLATFORM-RU.md` уже сделаны и облачный кластер работает. Тогда:

### Фаза D1. Закупка и подготовка железа (1–2 недели календарных)
- [ ] Заказать комплект (см. п. 1.1 / 1.2).
- [ ] Подготовить место: утепление, вентиляция, заземление, сетевой шкаф.
- [ ] UPS с NUT, протестировать graceful shutdown.

### Фаза D2. Локальная сеть и базовая ОС (1–2 дня)
- [ ] Настроить Mikrotik: VLAN, файрвол, WG-туннель к облаку.
- [ ] Поставить Ubuntu 22.04 LTS на узлы, базовый hardening.
- [ ] Поднять Pi-hole / AdGuard в Docker.

### Фаза D3. Дачный K3s (1 день)
- [ ] Поставить K3s на узлы (1 control + 1 worker минимум, 2+1 в идеале).
- [ ] Поставить cert-manager (DNS-01 через Cloudflare/Yandex), local-path provisioner, NFS provisioner для NAS-storage.
- [ ] Поставить Traefik для локального ingress.

### Фаза D4. Платформенные сервисы (2–3 дня)
- [ ] Vault standalone, инициализация, ACL.
- [ ] MinIO single-node tenant.
- [ ] Argo CD, подключение к Git (Gitea на даче или GitHub через mirror).
- [ ] Prometheus/Loki локально + remote_write в облачный Prometheus, чтобы видеть всё в одной Grafana.

### Фаза D5. Бэкап-партнёрство (1–2 дня)
- [ ] Настроить MinIO bucket replication: облачный velero-bucket → дачный.
- [ ] CronJob на даче для синка WAL-G архивов.
- [ ] Тест восстановления одного проекта в staging-namespace на даче.
- [ ] Документировать runbook'и.

### Фаза D6. Личные сервисы (по мере появления)
- [ ] Vaultwarden, Nextcloud, Gitea, Jellyfin, Home Assistant — каждый в своём ns, через Argo CD.
- [ ] Бэкапы личных сервисов в локальный MinIO.

## 8. Бюджет (итого, сверх облачной платформы)

### Капекс (одноразово)
- Стартовая конфигурация (см. 1.1): **~150 тыс. ₽**.
- Утепление/вентиляция шкафа: ~10–30 тыс. ₽ (зависит от состояния помещения).
- **Итого: ~160–180 тыс. ₽**.

### Опекс (₽/мес сверх облака)
| Статья | ₽/мес |
|---|---|
| Электричество (~150 Вт avg, ~108 кВт·ч) | ~700–1 200 |
| Интернет на даче (обычный, без статики) | 500–1 500 |
| VPS-фронтэнд для статус-страницы (Selectel мини) | 200 |
| Замена компонентов (амортизация диска/UPS-батареи) | ~500 |
| **Итого** | **~2 000–3 500** |

Сравнение полных опексов:
- **Только облако** (стартовая 3 VM): ~11 200 ₽/мес.
- **Гибрид** (облако + дача): ~13 000–15 000 ₽/мес.
- **Только дача** (если бы был серьёзный аплинк): ~7 000 ₽/мес, но рисково.

Прирост ~2 000 ₽/мес даёт: вторую географию для бэкапов, neograb-зону для личных сервисов, экспериментальную площадку, зеркало образов на случай блокировок.

## 9. Риски и mitigation

| Риск | Mitigation |
|---|---|
| Пропадание света/интернета на даче на сутки+ | Прод не страдает (он в облаке); UPS+NUT graceful shutdown; бэкапы догонят при восстановлении |
| Возгорание/затопление | Дымовой датчик с push в Telegram; страховка дома; критичные бэкапы дублируются в облако |
| Кража оборудования | Замок на стойке; шифрование дисков LUKS (ввод пароля при boot — минус, но для homelab ок); важное на ZFS-бекап в облаке |
| ZFS pool corruption | RAID-Z2 + scrub раз в месяц + бэкап датасетов в облачный S3 |
| WG-туннель отключился, не заметили | Алерт в Prometheus при отсутствии метрик с дачи >1 ч |
| Не успеваешь обслуживать (быт vs хобби) | Ограничить ответственность: личные сервисы могут лежать сутки, прод-клиенты — нет (поэтому они в облаке) |

## 10. Чек-лист «можно начинать»

- [ ] Облачная платформа из `PLATFORM-RU.md` запущена и стабильно работает ≥ 1 месяц.
- [ ] Решено помещение под стойку (утепление, питание, вентиляция).
- [ ] Заказано железо.
- [ ] Заведён домен и DNS-провайдер с API (Cloudflare/Yandex Cloud DNS).
- [ ] Настроен мониторинг доступности дачи извне (cloud Prometheus → blackbox-exporter → push до WG-туннеля).
- [ ] Готов runbook на случай отказа дачи и облака.
